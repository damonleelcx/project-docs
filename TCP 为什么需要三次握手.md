**TCP 为什么需要三次握手？**
三次握手（**Three-way Handshake**）是 TCP 在建立连接时用来**确认双方发送和接收能力正常，并同步初始序列号**的机制。

---

## 一、三次握手过程

客户端与服务器建立连接的流程：

```
Client                        Server
  | -------- SYN --------->     |
  |                             |
  | <------ SYN + ACK --------  |
  |                             |
  | -------- ACK --------->     |
```

### 第一次握手

客户端 → 服务器

发送 **SYN**（synchronize）包：

```
SYN = 1
seq = x
```

作用：

- 客户端告诉服务器：**我要建立连接**
- 同时发送自己的 **初始序列号 seq = x**

---

### 第二次握手

服务器 → 客户端

返回 **SYN + ACK**：

```
SYN = 1
ACK = 1
seq = y
ack = x + 1
```

作用：

- 确认收到客户端 SYN
- 告诉客户端 **服务器也可以建立连接**
- 同时发送服务器的 **初始序列号 y**

---

### 第三次握手

客户端 → 服务器

发送 **ACK**：

```
ACK = 1
ack = y + 1
```

作用：

- 确认收到服务器 SYN
- 至此双方都确认：
  - 自己的发送能力正常
  - 对方的接收能力正常

连接正式建立。

---

# 二、为什么不能两次握手？

如果只有 **两次握手**，会产生 **历史连接问题（旧 SYN 包）**。

### 场景

1. 客户端发送 SYN
2. 网络延迟很久才到服务器
3. 客户端已经放弃连接
4. 服务器收到 SYN 后直接建立连接

结果：

服务器会认为连接已经建立
但客户端 **根本不知道**

服务器会 **浪费资源**（连接泄漏）。

第三次握手可以解决：

- 客户端必须再确认一次
- 如果客户端没有回应 ACK
- 服务器就不会认为连接建立

---

# 三、核心目的

三次握手主要解决 **三个问题**

### 1 确认双方收发能力

双方都要确认：

```
我能发
你能收
你能发
我能收
```

---

### 2 同步初始序列号

TCP 是 **可靠字节流协议**，需要序列号保证：

- 不丢包
- 不乱序
- 可重传

---

### 3 防止旧连接请求

避免：

```
旧 SYN 包
网络延迟
导致错误连接
```

---

# 四、为什么不是四次？

其实 **理论上可以四次**，但 TCP 把 **SYN + ACK 合并了**：

原本可能是：

```
SYN
ACK
SYN
ACK
```

TCP优化为：

```
SYN
SYN+ACK
ACK
```

所以是 **三次**。

---

# 五、一个非常经典的面试总结

TCP 三次握手的目的：

```
1 确认双方收发能力
2 同步初始序列号
3 防止历史连接
```
